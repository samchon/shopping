# Frontend Architecture

## Overview

This is a Next.js 14 application built with the App Router, using the `@samchon/shopping-api` SDK to communicate with the backend. The architecture follows a clear separation of concerns with distinct layers.

## Architecture Layers

### 1. SDK Layer (`@samchon/shopping-api`)
- Auto-generated by Nestia from the backend
- Provides typed API functions and DTOs
- Supports simulation mode for testing

### 2. Adapter Layer (`src/adapters/`)
- **Purpose**: Abstract SDK calls and handle connection management
- **Responsibilities**:
  - Manage authentication tokens
  - Handle token refresh
  - Transform SDK errors into application errors
  - Batch related API calls
- **Example**:
  ```typescript
  // Instead of calling SDK directly:
  ShoppingApi.shoppings.customers.sales.index(connection, params)
  
  // Use the adapter:
  salesAdapter.list(params)
  ```

### 3. Domain Layer (`src/domain/`)
- **Purpose**: Normalize SDK types into application-specific models
- **Why**: Prevents SDK changes from cascading through the UI
- **Pattern**:
  ```typescript
  // SDK type
  type SDKSale = ShoppingApi.IShoppingSale;
  
  // Domain model
  interface Product {
    id: string;
    name: string;
    price: number;
    // Only fields the UI needs
  }
  
  // Transformation
  function sdkSaleToProduct(sale: SDKSale): Product {
    return {
      id: sale.id,
      name: sale.content.title,
      price: sale.price_range.lowest.real,
    };
  }
  ```

### 4. Hooks Layer (`src/hooks/`)
- **Purpose**: Business logic and state management
- **Tools**: React Query for server state
- **Pattern**:
  ```typescript
  export const useSales = (filters?: SaleFilters) => {
    return useQuery({
      queryKey: ['sales', filters],
      queryFn: () => salesAdapter.list(filters),
      select: data => data.results.map(sdkSaleToProduct),
    });
  };
  ```

### 5. UI Layer (`src/components/`, `src/app/`)
- **Purpose**: Presentation only
- **Dependencies**: Hooks for data, domain models for types
- **No direct SDK or adapter calls**

## Authentication Flow

### Flow Diagram
```
1. App loads → Check localStorage for token
2. If token exists:
   - Try to fetch current customer
   - If success → User is authenticated
   - If 401 → Try refresh token
     - If refresh success → New token stored
     - If refresh fail → Logout
3. If no token → Guest mode
```

### Token Management
- Access token: 3 hours validity
- Refresh token: 7 days validity
- Stored in localStorage (encrypted in production)
- Auto-refresh when expired

### Key Functions
```typescript
// lib/auth.ts
authStorage.get()        // Get stored auth state
authStorage.set()        // Store new auth state
authStorage.clear()      // Logout

// hooks/useAuthContext.tsx
useAuthContext()         // React context for auth state
```

## Main User Flows

### Customer: Browse & Purchase
1. Browse products (`/sales`)
2. View product details (`/sales/[id]`)
3. Add to cart (via modal or direct purchase)
4. Review cart (`/cart`)
5. Checkout:
   - Select delivery address
   - Apply coupons/mileages
   - Confirm order
6. Payment (external gateway integration)
7. Order confirmation (`/orders/[id]`)

### Customer: Account Management
- Login/Signup (`/login`, `/signup`)
- View orders (`/orders`)
- Manage wallet (`/wallet`)
- View coupons (`/coupons`)

### Seller: Product Management
- Dashboard (`/seller`)
- Create/Edit products (`/seller/sales/new`, `/seller/sales/[id]/edit`)
- View orders (`/seller/orders`)
- Manage deliveries (`/seller/deliveries`)

### Administrator: Governance
- Dashboard (`/admin`)
- Manage policies (`/admin/policies`)
- View analytics (`/admin/analytics`)
- Manage users (`/admin/users`)

## API Omissions

Intentionally NOT implementing (documented in `wiki/omissions.md`):

1. **External user authentication** - Out of scope for MVP
2. **Seller onboarding flow** - Sellers are pre-created
3. **Real-time notifications** - Polling-based for now
4. **Advanced search filters** - Basic search only
5. **Wishlist** - Not in SDK
6. **Product comparisons** - Not in SDK

## Testing Strategy

### E2E Tests (Playwright)
- SDK simulation mode enabled
- Tests run against frontend only (no backend required)
- Cover main user flows:
  - Product browsing
  - Cart operations
  - Order creation
  - Authentication

### Test Structure
```typescript
// tests/customer-browse.spec.ts
test('customer can browse and search products', async ({ page }) => {
  await page.goto('/sales');
  await expect(page.locator('.product-grid')).toBeVisible();
  
  await page.fill('#search', 'test product');
  await page.click('#search-button');
  
  await expect(page.locator('.product-card')).toHaveCountGreaterThan(0);
});
```

## Key Design Decisions

1. **Adapter Pattern**: Isolates SDK from UI, making refactoring easier
2. **Domain Models**: Prevents SDK coupling throughout the app
3. **React Query**: Centralized server state management
4. **shadcn/ui**: Consistent, accessible components
5. **SDK Simulation**: Enables frontend-only testing
6. **Next.js App Router**: Modern React Server Components architecture
